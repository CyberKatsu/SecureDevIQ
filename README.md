## Educational Rationale

AI code assistants (GitHub Copilot, Cursor, Claude) dramatically accelerate development — but they can confidently suggest code that contains subtle security vulnerabilities. These flaws aren't bugs; they're valid-looking, working code that happens to be exploitable.

SecureDevIQ trains developers to catch these flaws *before* they reach production through a deliberate-practice loop:

1. **Read** a realistic code snippet generated by Claude (the same kind of code an AI assistant would suggest)
2. **Analyse** it: identify the vulnerability type, its precise location, and why it's dangerous
3. **Receive immediate AI feedback**: a score out of 10, a breakdown of what you got right and wrong, and a concrete fix suggestion
4. **Track** your performance over time, revealing which vulnerability categories and difficulty levels need more work

### Vulnerability categories covered

| Category | Example |
|---|---|
| **Prompt Injection** | User input flows unsanitised into an LLM prompt |
| **Memory Issues** | Unbounded allocations, use-after-free in C extension wrappers |
| **Insecure Defaults** | Disabled TLS verification, weak ciphers, world-readable files |
| **Missing Sanitisation** | SQL injection, shell injection, path traversal, XSS |
| **Hardcoded Secrets** | API keys, passwords, tokens embedded in source code |

### Difficulty levels

- **Junior** — One clear, commonly-missed flaw (e.g. an obvious SQL injection)
- **Mid** — Subtle or context-dependent vulnerability (e.g. a race condition, TOCTOU)
- **Senior** — Multi-step exploit chain (e.g. SSRF enabling IMDS credential theft)

## Setup

### Prerequisites

- Docker + Docker Compose
- An Anthropic API key ([console.anthropic.com](https://console.anthropic.com/))

### 1. Clone and configure

```bash
git clone https://github.com/YOUR_USERNAME/securedeviq.git
cd securedeviq
cp .env.example .env
```

Open `.env` and set your key:

```dotenv
ANTHROPIC_API_KEY=sk-ant-...
SECRET_KEY=$(openssl rand -hex 32)   # generate a strong secret
```

### 2. Start with Docker Compose

```bash
docker compose up --build
```

This starts three services:
- **db** (Postgres 16) — waits for a health check before the backend starts
- **backend** (FastAPI/uvicorn) — runs `alembic upgrade head` then starts on port 8000
- **frontend** (Reflex) — serves the UI on port 3000

Open **http://localhost:3000** in your browser.

### 3. Running without Docker (local development)

```bash
# Terminal 1 — start Postgres (or point DATABASE_URL at an existing instance)
docker run -e POSTGRES_USER=securedeviq -e POSTGRES_PASSWORD=secret \
           -e POSTGRES_DB=securedeviq -p 5432:5432 postgres:16-alpine

# Terminal 2 — backend
cd backend
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
alembic upgrade head
uvicorn app.main:app --reload --port 8000

# Terminal 3 — frontend
cd frontend
pip install -r requirements.txt
reflex run
```

## How Challenge Generation Works

### Two-prompt architecture

Challenge generation and answer evaluation are kept as **entirely separate system prompts**, stored as constants in:

```
backend/app/prompts/
  generation.py   ← CHALLENGE_GENERATION_SYSTEM_PROMPT
  evaluation.py   ← CHALLENGE_EVALUATION_SYSTEM_PROMPT
```

**Why separate?**
- Each prompt can be iterated independently without touching business logic
- They are version-controlled alongside code — any change is reviewable in a PR
- The generation prompt never "leaks" the reference answer to the evaluation step (the evaluator reads it from the DB, not from the generation response)

### Generation flow

```
POST /api/challenges/generate
        │
        ▼
  Claude (generation prompt)
        │
        ▼
  JSON parsed → GeneratedChallenge dataclass
        │
        ▼
  Challenge row inserted into PostgreSQL
  (reference_explanation stored; NOT returned to the frontend)
        │
        ▼
  ChallengeResponse returned (no reference_explanation — ground truth stays secret)
```

### Evaluation flow

```
POST /api/submissions
        │
        ▼
  Challenge fetched from DB (includes reference_explanation)
        │
        ▼
  Claude (evaluation prompt)
  ← receives: code_snippet + reference_explanation + user_answer
        │
        ▼
  JSON parsed → ScoringResult (Pydantic-validated: score 0–10, lists, strings)
        │
        ▼
  Submission row persisted with all scoring fields
        │
        ▼
  SubmissionResponse returned to Reflex frontend
```

### Caching strategy

Generated challenges are **cached in PostgreSQL** after the first generation. This means:
- Different users can attempt the same challenge without triggering duplicate API calls
- Scoring baselines are consistent (the same `reference_explanation` is always used)
- The `GET /api/challenges` endpoint lists existing challenges so users can filter by language/difficulty

## Running Tests

```bash
cd backend
pip install -r requirements.txt
pytest tests/ -v
```

Tests use **in-memory SQLite** (via `aiosqlite`) — no Postgres instance required. The Anthropic client is fully mocked, so no API key is needed.

### Test coverage

| File | What it tests |
|---|---|
| `tests/test_scoring.py` | `ScoringResult` Pydantic model validation — bounds, required fields, rounding |
| `tests/test_generation.py` | `generate_challenge()` — JSON parsing, fence stripping, missing fields, enum conversion |
| `tests/test_challenges_api.py` | `GET/POST /api/challenges` routes — auth guards, 422 validation, persistence, filtering |

## Security Notes

- `reference_explanation` (ground truth) is **never returned** to the frontend — only stored in the DB and passed to the evaluation prompt
- JWTs use HS256; rotate `SECRET_KEY` in production
- Set a real `SECRET_KEY` via environment variable — never rely on the default
- The Anthropic API key is only accessed server-side; the frontend talks only to the FastAPI backend

## Contributing

Pull requests welcome. To add a new vulnerability category:
1. Add it to `VulnCategory` enum in `backend/app/models/__init__.py`
2. Update `CHALLENGE_GENERATION_SYSTEM_PROMPT` to describe it
3. Add it to `CATEGORIES` in `frontend/securedeviq/pages/challenge.py`
4. Run `alembic revision --autogenerate -m "add new category"` to create a migration
